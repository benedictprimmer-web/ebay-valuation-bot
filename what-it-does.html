eBay Valuation Bot

What it actually does, in plain terms — and the data-source choice still open.
Read-only Never bids 18 tests passing Private-seller fees

In one sentence

Every few minutes it looks at graded football-card auctions ending soon, works out what each card is really worth from other listings of the exact same card, subtracts every cost, and if there's a safe profit it WhatsApps you the card and the most you should bid. You place the bid. It never spends a penny on its own.

The clever bit: it values a range, then bids against the cautious end

A single "worth £80" hides how sure the bot is. So instead it builds a spread from the comps and bids against the low end. Tight, plentiful comps → confident → near-full bid. Few or scattered comps → the number shrinks on its own, or it stays quiet. Uncertainty isn't a warning label, it's baked into the price. "Is valuation.py code?" — yes. Here's every file in English

FileWhat it does
valuation.pyThe clever bit above. Turns comps into a price range + a confidence score, and the cautious value the bot bids against.
fees.pyEvery cost between winning and reselling. Now defaults to your £0 private-seller fees, with a business switch. Solves the max bid.
threshold.pyThe gatekeeper. Enough comps? Tight enough? Clears the 25% / £15 floors? Under the £50 cap? Builds the verdict per card.
ebay_client.pyWhere the listings come from. A mock source (for testing now) and the live eBay one, behind the same plug — so swapping the data source is a small job.
alert.pySends the WhatsApp via CallMeBot. One web request.
store.pyRemembers what it already alerted on, and logs every prediction so you can check it against what actually sold.
pipeline.pyRuns the seven steps in order.
run.pyThe button. python run.py --mode mock tests it with no keys.

The 18 passing tests run on built-in fake listings, so they prove the maths and the logic without needing eBay keys. The live data feed is the one piece still to wire — that's the choice below.

Where the data comes from — the choice that's still open

The original plan used eBay's official API for active listings (what people are asking). That needs the developer account you found annoying. There are two other routes, and one is genuinely better than what we planned.

Planned

Official eBay API

Active listings (asking prices)
  • Free, fully allowed
  • Already built
  • Needs the developer-account approval (slow, can be denied)
  • Asking ≠ sold, so we guess a ratio
My recommendation

Third-party API

Real sold prices (RapidAPI)
  • No eBay developer account at all
  • Actual sold prices — the true signal, no ratio guess
  • Live in an hour; a few £/month
  • Small cost + relies on their service staying up
Avoid

Scraping

Reading eBay's web pages
  • Free, instant, sold data
  • Breaks eBay's rules
  • IP bans / CAPTCHAs
  • Breaks whenever eBay changes its site
The point Gemini got right: sold prices beat asking prices. A third-party sold-data API gets you that and skips the developer hassle. The whole valuation engine, fees and alerts already work — only this data feed swaps in. The trade-off is a small monthly cost and depending on someone else's service.
